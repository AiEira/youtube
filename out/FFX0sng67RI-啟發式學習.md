# 啟發式學習（HL）：當 AI 不再訓練神經網絡，只改代碼

> Source: YouTube · 為什麼叫QQ · Speaker: 翁家翌（OpenAI Post-Training 研究工程師）
> Video: https://youtu.be/FFX0sng67RI
> Date: 2026-05-17

---

## 一個瘋狂的實驗

一個 AI，不訓練神經網絡，不更新任何權重。就只是看失敗錄像、改代碼、加測試——把 Atari 打磚塊遊戲推到 **864 分的理論最高分**。

提出這個想法的是 OpenAI Post-Training 團隊的研究工程師 **翁家翌**（清華本科、CMU 碩士、TSAIL / CAI Lab 出身）。他是 ChatGPT RLHF 基礎設施的參與者，也是開源 RL 庫 Tianshou 和超快速環境執行引擎 EnvPool 的作者。

這個發現源於他在業餘時間維護 EnvPool 時的一個小需求：寫一些便宜的啟發式規則來測試遊戲環境的正確性，省得每次跑 CI 都要動用昂貴的神經網絡。於是他讓 Codex 模型去寫一個完全不依賴神經網絡的純代碼版本。

結果事情的發展完全超出了預期。

---

## 純代碼策略跑出了什麼成績？

- **Atari 打磚塊**：387 → 507 → 839 → **864 分（理論最高分）**
- **MuJoCo 四足機器人控制**：純 Python 程序先學會節律步態，最後拿到 6000+ 分，直接殺入深度強化學習結果的量級
- **MuJoCo Cheetah 跑步**：靠可解釋的步態和姿態規則，5 局複測均值 **11,836.7 分**
- **VizDoom 第一人稱射擊**：只用 OpenCV 截屏幕視覺，不訓練任何網絡，平均 **557 分**
- **Atari 57 遊戲全測試**：固定環境交互步數下，純代碼策略的中位數得分在 100 萬步附近**遠遠甩開 PPO 這類主流深度強化學習算法**

而這一切——**沒有反向傳播，沒有梯度下降。只有一行行不斷生長的代碼。**

## 啟發式學習（HL）是什麼？

翁嘉毅把這個過程定義為 **啟發式學習（Heuristic Learning，HL）**。

| | 深度強化學習（DRL） | 啟發式學習（HL） |
|---|---|---|
| 更新對象 | 神經網絡參數 | **軟件結構（代碼）** |
| 策略形式 | 權重矩陣 | 狀態機、控制邏輯、MPC |
| 反饋來源 | Reward 數字 | 測試用例、環境日誌、失敗視頻 |
| 更新方式 | 梯度下降 | Coding agent 直接改代碼 |
| 可解釋性 | 黑盒 | 一行行可讀 |
| 樣本效率 | 數百萬–數十億步 | 可能數千步就見顯著改進 |

在 HL 中，被長期維護的對象被稱為 **啟發式系統（Heuristic System，HS）**。它不是一個孤立的策略文件，而是包含了狀態表示、反饋入口、實驗記錄、測試用例和記憶機制的**完整軟件系統**。

---

## HL 的三個核心特質

### 1. 極強的可解釋性

神經網絡是黑盒。HL 的代碼策略——你可以一行行讀懂它在做什麼。對於需要可信度的場景（自動駕駛、醫療決策），這意義重大。

### 2. 恐怖的樣本效率

一次有效的代碼更新，策略就能發生質的飛躍，根本不需要吊著學習率慢慢爬坡。DRL 需要數百萬甚至數十億的環境交互才能收斂；HL 可能數千步就能看到顯著改進。

### 3. 對抗災難性遺忘的新武器：可回歸驗證

舊的能力不需要全靠模型自己死記硬背，而是可以被固化——化成回歸測試、固定種子的回放、或明確寫下來的失敗方向。新代碼如果破壞了舊場景，測試用例立刻報警。

> 這就像是給系統裝上了一個**記憶外骨骼**，確保你不會在前进的同時丟掉過去。

---

## 為什麼以前沒人這麼做？

主流認知一直覺得專家系統和手寫規則走不遠——**維護成本太高了**。人類程序員去維護一套複雜的規則：今天加 if 修好一個 bug，明天發現另一個場景崩了，後天再打個補丁，大後天這堆代碼就成了誰也不敢碰的屎山。

**但現在局面變了。Coding agent 的出現，徹底改變了啟發式規則的維護成本曲線。**

以前靠人力維護，專家系統就像工業革命前的手工紡紗，規模稍微一大人就被壓垮。而現在的 coding agent，就像一台不知疲倦的紡織機——它能持續讀日誌、看回放、改代碼、跑測試。它變成了一條**輸送智力的營養管道**，能夠自動化地澆灌一個啟發式系統，讓它自己迭代進化。

過去因為維護成本太高而不值得寫的規則，現在突然變得值得長期擁有了。

> **這就是範式轉移的本質：不是技術本身變強了，而是生產成本曲線變了。**

---

## 持續學習：從參數死胡同到工程問題

「持續學習」（continual learning）這個難題，一直被困在「如何更新參數」的死胡同裡。HL 告訴我們：**我們可以把它變成一個工程問題——如何維護一個持續吸收反饋的軟件系統。**

一個健康的啟發式系統必須具備**吸收反饋和壓縮歷史**的能力。它不能只增長不壓縮，否則依然會變成連 agent 自己都維護不動的屎山。它需要把一堆局部的補丁重構成更簡單、更優雅的代碼表示。

**這就像是代碼的新陳代謝。**

這就引出了一個核心概念：**耦合複雜度**。一個系統能走多遠，取決於 coding agent 能處理多複雜的耦合關係。良好的模塊化、完善的測試覆蓋、清晰的日誌——都能有效降低耦合複雜度。這不再只是單純的算法比拼，更是一場**系統工程的較量**。

---

## HL 能取代神經網絡嗎？

不能完全取代。在複雜的視覺感知和長程泛化上，純代碼依然力不從心——你不可能用 if-else 寫出一個 ImageNet 分類器。

最激動人心的未來是**兩者的結合**。比如在機器人控制中：

```
┌─────────────────────────────────────┐
│  LLM Agent（大腦）                   │
│  高層反饋，周期性吸收 HL 數據更新自己  │
├─────────────────────────────────────┤
│  HL 層（最新數據處理）                │
│  安全邊界、局部恢復、代碼迭代          │
├─────────────────────────────────────┤
│  專用淺層神經網絡（底層感知）          │
│  快速狀態估計、視覺處理               │
└─────────────────────────────────────┘
```

這可以繼續拆成層級結構：關節級 HL → 肢體級 HL → 全身平衡 HL → 任務級 HL。低層負責安全和低延遲控制，中層負責步態和接觸，高層負責任務恢復和長期記憶。

Coding agent 不一定直接懂得走路——它更像**插進系統裡的更新管線**：持續把失敗視頻、傳感器流、仿真結果、測試結果餵進系統，再把反饋改寫成代碼、參數保護規則和 memory。

---

## 核心金句

> **「凡是可以被持續迭代的，都開始能被解決。」**

從梯度更新到代碼迭代，AI 正在用工程化的方式重新定義學習的本質。這不僅僅是一個算法的突破，更是一個思維方式的轉變。
